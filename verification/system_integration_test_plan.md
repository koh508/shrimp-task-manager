# 온유 시스템 개선 버전 통합 테스트 계획

## 1. 문서 개요

이 문서는 온유 시스템의 개선된 로직이 성공적으로 통합되고, 시스템의 목표를 달성했는지 최종적으로 검증하기 위한 통합 테스트 계획을 상세히 기술합니다. 본 계획은 시스템의 핵심 기능, 성능, 안정성 및 예외 처리 능력을 종합적으로 평가하여, 개선된 시스템이 프로덕션 환경에 배포될 준비가 되었음을 확인하는 데 목적이 있습니다.

## 2. 테스트 범위

개선된 온유 시스템의 `main.py` 스크립트를 중심으로, 통합된 모든 로직과 컴포넌트 간의 상호작용을 테스트합니다. 이는 다음을 포함합니다:
*   핵심 비즈니스 로직의 정확성 및 완전성
*   외부 시스템 또는 서비스와의 연동 (API 호출 등)
*   데이터 처리 및 저장 메커니즘
*   예외 상황 및 오류 처리
*   시스템 성능 및 응답 시간 (기본적인 검증)

## 3. 테스트 환경 설정

### 3.1. 전제 조건

*   **운영체제:** Linux (Ubuntu 20.04 이상 권장) 또는 macOS
*   **Python 버전:** Python 3.9 이상
*   **의존성 설치:** `pip install -r requirements.txt` 명령을 통해 모든 필요한 라이브러리 설치 완료.
*   **환경 변수 설정:**
    *   `API_KEY_EXTERNAL_SERVICE`: 외부 서비스 연동에 필요한 API 키
    *   `DB_CONNECTION_STRING`: 데이터베이스 연결 문자열
    *   `LOG_LEVEL`: 로깅 레벨 (예: `INFO`, `DEBUG`)
    *   `CONFIG_PATH`: 시스템 설정 파일 경로 (필요시)
    *   **설정 방법:** 테스트 실행 전 `.env` 파일에 정의하거나, 쉘 환경 변수로 설정합니다.

### 3.2. 데이터 준비

*   **실제 데이터 사용:** 가능한 경우, 프로덕션 환경과 유사한 형태의 실제 데이터를 익명화하여 사용합니다.
*   **테스트 데이터셋:** 특정 시나리오를 검증하기 위한 고유한 테스트 데이터셋을 준비합니다. (예: 정상 케이스, 경계 케이스, 오류 유발 케이스)
*   **초기 상태:** 테스트 시작 전, 데이터베이스 또는 파일 시스템의 상태를 정의된 초기 상태로 복원합니다. (예: 특정 테이블 비우기, 초기 데이터 삽입)

## 4. 테스트 시나리오 및 상세 계획

### 4.1. 시나리오 1: 핵심 로직 실행 및 결과 검증

**목표:** 개선된 시스템의 핵심 비즈니스 로직이 예상대로 동작하고, 정확한 최종 결과를 생성하는지 확인합니다.

*   **테스트 케이스 ID:** INTEGRATION-CORE-001
*   **설명:** `main.py`를 실행하여 시스템의 주요 데이터 처리 흐름을 검증하고, 최종 출력이 올바른지 확인합니다.
*   **사전 조건:**
    *   모든 환경 변수가 올바르게 설정되어 있습니다.
    *   테스트 데이터셋 `data_set_A.json`이 `data/` 디렉토리에 존재합니다.
    *   데이터베이스가 초기 상태로 설정되어 있습니다.
*   **테스트 단계:**
    1.  `python main.py --input data/data_set_A.json --output_path results/output_A.json` 명령을 실행합니다.
    2.  스크립트 실행이 완료될 때까지 기다립니다.
    3.  `results/output_A.json` 파일이 생성되었는지 확인합니다.
    4.  `results/output_A.json` 파일의 내용을 미리 정의된 `expected_output_A.json`과 비교합니다.
    5.  데이터베이스의 특정 테이블 (`processed_data_table`)에 데이터가 올바르게 저장되었는지 확인합니다.
*   **예상 결과:**
    *   `main.py` 스크립트가 오류 없이 성공적으로 종료됩니다.
    *   `results/output_A.json` 파일이 생성되며, 그 내용이 `expected_output_A.json`과 정확히 일치합니다.
    *   `processed_data_table`에 100개의 레코드가 삽입되었으며, 각 레코드의 `status` 필드가 'COMPLETED'로 되어 있습니다.
*   **성공 기준:**
    *   스크립트 종료 코드 0.
    *   출력 파일 내용 일치율 100%.
    *   데이터베이스 삽입 레코드 수 및 필드 값 일치.
*   **테스트 데이터:** `data_set_A.json` (정상적인 입력 데이터 100개 포함), `expected_output_A.json`
*   **수행 방법:** 수동으로 명령 실행 후 파일 비교 및 DB 쿼리 확인. 자동화된 스크립트 (예: `pytest` 또는 `unittest` 기반)를 사용하여 출력 파일 및 DB 상태를 검증.

### 4.2. 시나리오 2: 외부 서비스 연동 및 예외 처리

**목표:** 외부 서비스와의 연동이 올바르게 이루어지고, 외부 서비스 오류 발생 시 시스템이 적절하게 예외를 처리하는지 확인합니다.

*   **테스트 케이스 ID:** INTEGRATION-EXTERNAL-001
*   **설명:** 외부 API 호출이 포함된 로직을 실행하고, API 호출 성공 및 실패 시 시스템의 동작을 검증합니다.
*   **사전 조건:**
    *   `API_KEY_EXTERNAL_SERVICE`가 유효하게 설정되어 있습니다.
    *   테스트 데이터셋 `data_set_B_external.json`이 `data/` 디렉토리에 존재합니다.
*   **테스트 단계 (성공 케이스):**
    1.  `python main.py --input data/data_set_B_external.json --output_path results/output_B_external.json` 명령을 실행합니다.
    2.  스크립트 실행이 완료될 때까지 기다립니다.
    3.  로그 파일 (`system.log`)에 외부 API 호출 성공 메시지가 기록되었는지 확인합니다.
    4.  `results/output_B_external.json` 파일에 외부 API 응답이 올바르게 반영되었는지 확인합니다.
*   **예상 결과 (성공 케이스):**
    *   스크립트가 오류 없이 성공적으로 종료됩니다.
    *   로그에 `[INFO] External API call successful for item ID: XXX` 메시지가 다수 기록됩니다.
    *   출력 파일에 외부 서비스로부터 받은 `external_status: 'SUCCESS'` 필드가 포함되어 있습니다.
*   **성공 기준 (성공 케이스):**
    *   스크립트 종료 코드 0.
    *   로그에 성공 메시지 기록 확인.
    *   출력 파일에 외부 서비스 응답 필드 정확성 확인.
*   **테스트 데이터:** `data_set_B_external.json` (외부 API 호출이 필요한 데이터 50개 포함)
*   **수행 방법:** 실제 외부 서비스에 연결하여 테스트.

*   **테스트 케이스 ID:** INTEGRATION-EXTERNAL-002
*   **설명:** 외부 API 호출 실패 시 시스템이 예외를 적절히 처리하고, 재시도 로직이 있다면 이를 검증합니다.
*   **사전 조건:**
    *   `API_KEY_EXTERNAL_SERVICE`를 의도적으로 잘못된 값으로 설정하거나, 외부 서비스가 응답하지 않도록 네트워크를 차단합니다.
    *   테스트 데이터셋 `data_set_B_external.json`이 `data/` 디렉토리에 존재합니다.
*   **테스트 단계 (실패 케이스):**
    1.  `API_KEY_EXTERNAL_SERVICE` 환경 변수를 `INVALID_KEY`로 설정합니다.
    2.  `python main.py --input data/data_set_B_external.json --output_path results/output_B_external_error.json` 명령을 실행합니다.
    3.  스크립트 실행이 완료될 때까지 기다립니다.
    4.  로그 파일 (`system.log`)에 외부 API 호출 실패 메시지 및 예외 처리 로직이 기록되었는지 확인합니다.
    5.  출력 파일 (`results/output_B_external_error.json`)에 실패한 항목에 대한 오류 정보가 올바르게 기록되었는지 확인합니다.
*   **예상 결과 (실패 케이스):**
    *   스크립트가 오류 없이 종료되거나, 정의된 예외 처리 로직에 따라 특정 오류 코드로 종료됩니다.
    *   로그에 `[ERROR] External API call failed for item ID: XXX, Error: Authentication Failed` 또는 `[WARNING] Retrying external API call for item ID: XXX`와 같은 메시지가 기록됩니다.
    *   출력 파일에 `external_status: 'FAILED'` 및 `error_message: '...'` 필드가 포함되어 있습니다.
*   **성공 기준 (실패 케이스):**
    *   스크립트가 비정상 종료되지 않고, 정의된 예외 처리 흐름을 따릅니다.
    *   로그에 적절한 오류/경고 메시지 기록 확인.
    *   출력 파일에 오류 정보가 정확하게 반영됩니다.
*   **테스트 데이터:** `data_set_B_external.json`
*   **수행 방법:** 환경 변수 조작 또는 네트워크 차단을 통해 외부 서비스 오류를 시뮬레이션.

### 4.3. 시나리오 3: 성능 및 자원 사용량 (기본 검증)

**목표:** 대량의 데이터를 처리할 때 시스템의 기본적인 성능과 자원 사용량을 확인합니다.

*   **테스트 케이스 ID:** INTEGRATION-PERF-001
*   **설명:** 대량의 입력 데이터를 사용하여 `main.py`를 실행하고, 실행 시간 및 메모리 사용량을 측정합니다.
*   **사전 조건:**
    *   모든 환경 변수가 올바르게 설정되어 있습니다.
    *   대량 테스트 데이터셋 `data_set_C_large.json` (예: 10,000개 이상의 레코드)이 `data/` 디렉토리에 존재합니다.
*   **테스트 단계:**
    1.  시스템 모니터링 도구 (예: `htop`, `top`, `psutil` 기반 스크립트)를 준비하여 CPU 및 메모리 사용량을 모니터링합니다.
    2.  `time python main.py --input data/data_set_C_large.json --output_path results/output_C_large.json` 명령을 실행합니다.
    3.  스크립트 실행이 완료될 때까지 기다립니다.
    4.  `time` 명령의 출력 (실행 시간)을 기록합니다.
    5.  모니터링 도구를 통해 기록된 최대 CPU 및 메모리 사용량을 확인합니다.
*   **예상 결과:**
    *   스크립트가 5분 이내에 성공적으로 종료됩니다.
    *   최대 CPU 사용률이 80%를 초과하지 않습니다.
    *   최대 메모리 사용량이 2GB를 초과하지 않습니다.
*   **성공 기준:**
    *   정의된 시간 내에 스크립트 성공적 종료.
    *   CPU 및 메모리 사용량이 허용 가능한 임계값 이내.
*   **테스트 데이터:** `data_set_C_large.json` (대량의 입력 데이터)
*   **수행 방법:** `time` 명령 및 시스템 모니터링 도구를 사용하여 측정. 필요시 `locust`와 같은 부하 테스트 도구를 활용하여 더 정밀한 성능 테스트를 수행할 수 있습니다.

### 4.4. 시나리오 4: 데이터 영속성 및 무결성

**목표:** 시스템이 데이터를 올바르게 저장하고, 저장된 데이터의 무결성이 유지되는지 확인합니다.

*   **테스트 케이스 ID:** INTEGRATION-DB-001
*   **설명:** `main.py` 실행 후 데이터베이스에 저장된 데이터가 예상되는 형태와 값으로 정확히 일치하는지 검증합니다.
*   **사전 조건:**
    *   `DB_CONNECTION_STRING`이 유효하게 설정되어 있습니다.
    *   데이터베이스가 초기 상태로 설정되어 있습니다.
    *   테스트 데이터셋 `data_set_D_db.json`이 `data/` 디렉토리에 존재합니다.
*   **테스트 단계:**
    1.  `python main.py --input data/data_set_D_db.json --output_path results/output_D_db.json` 명령을 실행합니다.
    2.  스크립트 실행이 완료될 때까지 기다립니다.
    3.  데이터베이스 클라이언트 (예: `psql`, `mysql` 클라이언트)를 사용하여 `processed_data_table`에서 데이터를 조회합니다.
    4.  조회된 데이터의 레코드 수, 특정 필드의 값, 그리고 데이터 타입이 예상과 일치하는지 확인합니다.
*   **예상 결과:**
    *   스크립트가 오류 없이 성공적으로 종료됩니다.
    *   `processed_data_table`에 `data_set_D_db.json`의 각 항목에 해당하는 50개의 레코드가 정확히 삽입되어 있습니다.
    *   각 레코드의 `timestamp` 필드가 유효한 날짜/시간 형식이며, `processed_value` 필드가 계산된 값과 일치합니다.
*   **성공 기준:**
    *   스크립트 종료 코드 0.
    *   데이터베이스에 삽입된 레코드 수 및 내용의 정확성.
    *   데이터 타입 및 형식의 일관성.
*   **테스트 데이터:** `data_set_D_db.json` (데이터베이스 저장 로직을 포함하는 데이터 50개)
*   **수행 방법:** `main.py` 실행 후 SQL 쿼리를 통해 데이터베이스 직접 검증.

## 5. 테스트 수행 방법

*   **자동화된 스크립트:** 가능한 모든 테스트 시나리오는 Python의 `pytest` 또는 `unittest` 프레임워크를 사용하여 자동화된 테스트 스크립트로 구현합니다. 이는 반복적인 테스트 수행과 결과의 일관성을 보장합니다.
*   **수동 검증:** 자동화하기 어려운 특정 시나리오 (예: 복잡한 로그 분석, UI가 있는 경우의 수동 확인)는 수동으로 수행하며, 그 결과를 상세히 기록합니다.
*   **CI/CD 통합:** 자동화된 테스트는 CI/CD 파이프라인에 통합하여 코드 변경 시마다 자동으로 실행되도록 설정합니다.
*   **로깅:** `main.py`는 상세한 로깅을 수행해야 하며, 테스트 중 발생하는 모든 이벤트, 경고, 오류는 `system.log` 파일에 기록되어야 합니다. 테스트 결과 분석 시 이 로그 파일을 활용합니다.
