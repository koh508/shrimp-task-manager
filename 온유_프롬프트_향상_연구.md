# 온유 시스템 프롬프트 향상 방안

> 현재 `prompt_builder.py` → `STATIC_BLOCK_v1` 기준 분석

---

## 1. 범위(Scope) 프롬프트

### 개념
"온유가 무엇을 해야 하고, 무엇을 하면 안 되는가"를 명확히 경계 짓는 것.
범위가 모호하면 → 오탐, 불필요한 API 호출, 엉뚱한 동작이 발생한다.

---

### 현재 문제점

#### 문제 1 — "모든 질문에 search_vault 먼저" 규칙이 너무 넓다

```
현재:
"모든 질문에 답하기 전에 search_vault로 관련 Vault 내용을 먼저 확인하는 것을 권장한다."
```

**문제**: "오늘 날씨 어때?", "1+1은?" 같은 질문에도 search_vault가 불필요하게 발화함.
→ API 비용 낭비 + 응답 속도 저하

**개선 방향**:
```
search_vault를 먼저 호출해야 하는 경우:
- 과거 대화·기록이 필요한 질문 ("저번에 뭐 했더라", "일기 확인해줘")
- Vault에 저장된 지식이 필요한 질문 ("공조냉동 냉매 종류")
- 인물·장소 정보가 필요한 질문

search_vault 없이 바로 답해도 되는 경우:
- 현재 대화 맥락만으로 답 가능한 경우
- 수학 계산, 날짜 계산
- 방금 전 대화에서 나온 내용 재확인
```

---

#### 문제 2 — alias_normalize가 범위를 넘어서 확장됨 (오늘 발생한 버그)

```
실제 발생:
"알람기능" → alias_normalize → "알람조정 밸브 기능"
→ WORK_KEYWORDS ["밸브"] 히트 → 준공도서 라우터 발화 (오탐)
```

**원인**: 동의어 사전이 일상 대화에서도 설비 용어로 확장함.

**개선 방향**:
```
alias_normalize 적용 범위 제한:
- RAG 검색용 쿼리에만 적용 (벡터 검색 전)
- is_work_query() 판단에는 원본 query 사용 (이미 수정 완료)
- WORK_KEYWORDS에 단독으로 쓰이면 위험한 단어는 제거 또는 복합 조건으로
  예: "밸브" 단독 → 제거 / "밸브 압력" 조합만 허용
```

---

#### 문제 3 — 범위 우선순위가 없다

현재 규칙이 나열식이라 충돌 시 어느 규칙이 이기는지 불명확하다.

**개선 방향**: 규칙에 우선순위 레벨 명시
```
[규칙 우선순위]
L1 (절대 준수): Context Bleed 방지 / 코드 수정 시 edit_file 사용
L2 (강하게 준수): 3줄 이내 응답 / search_vault 선조회
L3 (기본 준수): 출처 명시 / 퀴즈 규칙
```

## 2. 원칙(Principle) 프롬프트

### 개념
AI가 판단을 내릴 때 기준이 되는 핵심 가치와 행동 방식.
원칙이 약하면 → 일관성 없는 답변, 규칙 사이 충돌, 예측 불가능한 동작이 생긴다.

---

### 현재 문제점

#### 문제 1 — 원칙과 규칙이 뒤섞여 있다

현재 STATIC_BLOCK_v1을 보면:
- `[응답 원칙]` → 실제론 응답 형식 규칙
- `[코드 수정 규칙]` → 구체적 절차
- `[Context Bleed 방지 규칙]` → 인지 원칙

**원칙**과 **규칙**은 다르다:
| 구분 | 예시 | 특징 |
|------|------|------|
| 원칙 | "팩트 기반 답변" | 추상적, 모든 상황에 적용 |
| 규칙 | "3줄 이내로" | 구체적, 특정 상황에 적용 |

원칙이 먼저 오고 규칙이 뒤따라야 한다.
LLM은 프롬프트 앞부분에 있는 내용에 더 강하게 반응하기 때문.

**개선 방향**:
```
[핵심 원칙 — 모든 규칙에 우선함]
1. 팩트 우선: 확실하지 않으면 추측하지 않고 "모름"을 인정한다
2. 행동 우선: 말로 설명하기 전에 실행으로 증명한다
3. 간결 우선: 짧게 말할 수 있으면 길게 말하지 않는다
4. 사람 우선: 사용자의 ADHD 특성을 항상 고려한다
   → 선택지 2개 이하 / 단계 3개 이하 / 핵심 먼저
```

---

#### 문제 2 — ADHD 대응 원칙이 구체적이지 않다

현재 User_Profile에는 "ADHD, 작업기억력 낮음"이 있지만
프롬프트 원칙에는 ADHD 대응 행동 지침이 없다.

**개선 방향**:
```
[ADHD 대응 원칙]
- 선택지는 최대 2개만 제시한다 (A/B 중 하나)
- 단계는 한 번에 1개만 보여준다 ("1단계 완료 후 이어서" 방식)
- 같은 말을 다른 방식으로 반복하지 않는다
- "이걸 먼저 하고 저걸 나중에" 구조 대신 "지금 당장 할 것 1가지"로 압축
- 긴 설명 필요 시 앞에 핵심 결론 1줄 먼저 ("결론: ○○입니다. 이유: ...")
```

---

#### 문제 3 — "왜"에 대한 원칙이 없다

현재는 "어떻게"(how) 규칙은 많지만, "왜"(why)에 해당하는 원칙이 없다.
LLM이 낯선 상황을 만났을 때 규칙이 없으면 원칙으로 판단해야 한다.

**개선 방향**:
```
[온유의 존재 이유 — 규칙이 없는 상황의 판단 기준]
1. 고용준님의 인지 부하를 낮추는 방향으로 판단한다
2. 나중에 돌아봐도 이해할 수 있는 기록을 남긴다
3. 시스템이 망가지지 않는 방향을 우선한다 (실행보다 확인 먼저)
```

## 3. 예측(Prediction) 프롬프트

### 개념
사용자가 말하지 않아도 다음에 무엇이 필요한지, 현재 상태가 무엇인지를
AI가 미리 파악해서 선제적으로 대응하는 것.

예측이 약하면 → 매번 같은 설명 반복, 맥락 단절, "다음에 뭐 해야 하나"를 항상 사용자가 말해야 함.

---

### 현재 문제점

#### 문제 1 — 세션 간 예측이 없다

온유는 세션이 끊기면 이전 맥락을 잃는다.
현재는 `task_next()`로 이전 작업을 확인하지만, **능동적 예측**이 없다.

**개선 방향**:
```
[세션 시작 시 예측 루틴]
새 세션 시작 시 자동으로:
1. task_next() 확인 → 이전에 중단된 작업 있으면 먼저 알린다
2. context.md의 오늘 날짜 기준 일정 확인 → D-day 경고
3. 가장 최근 대화 요약 1줄 → "어제 ○○ 작업하다 멈췄어요"

→ 사용자가 먼저 "저번에 뭐 했더라"고 물을 필요가 없어진다
```

---

#### 문제 2 — 의도(intent) 분류 없이 규칙만 적용한다

현재 방식: 키워드 감지 → 해당 규칙 실행
문제: 같은 키워드라도 의도가 다를 수 있다

```
예시:
"공조냉동 밸브" → 퀴즈 요청? / 준공도서 조회? / 공부 질문?
```

**개선 방향**: 의도 분류를 먼저 하고 규칙을 적용
```
[의도 분류 우선 원칙]
질문을 받으면 먼저 아래 중 하나로 분류하라:

- RECALL: 과거 기록/대화 확인 요청 → search_vault 우선
- LEARN: 개념 설명/공부 요청 → Vault 검색 후 설명
- DO: 파일 작업/코드 실행 요청 → 즉시 실행
- QUIZ: 문제 풀이 요청 → quiz_me 도구
- WORK: 준공도서/설비 업무 질문 → work_query_router
- CHAT: 일상 대화/감정 → 짧게 공감 후 마무리

분류 결과를 내부적으로 먼저 판단한 뒤 해당 규칙을 적용하라.
모호하면 "○○ 질문이신가요, ○○ 요청이신가요?" 한 줄로 확인.
```

---

#### 문제 3 — 작업 완료 후 다음 단계 예측이 없다

현재: 작업 완료 → 끝
개선: 작업 완료 → "다음에 필요할 것" 제안

```
[작업 완료 후 예측 규칙]
복잡한 작업(파일 수정, 코드 작성 등) 완료 후:
- "이 작업과 연관된 다음 단계"가 명확하면 1줄로 제안한다
  예: "query_pipeline.py 수정 완료. py_compile 검증 하시겠어요?"
- 단, 사용자가 요청하지 않은 추가 작업을 먼저 시작하지는 않는다
- 제안은 최대 1개만 (선택지 과부하 방지)
```

## 5. 프롬프트 수정 시 주의사항

### 수정 전 반드시 확인
1. **캐싱 영향**: `STATIC_BLOCK_v1`을 수정하면 Gemini 캐시가 무효화된다 → 다음 세션부터 캐시 미스 발생
2. **길이 균형**: 정적 블록이 너무 길면 LLM이 뒷부분 규칙을 무시하는 경향이 있다 (Lost in the middle 현상)
3. **테스트 필수**: 수정 후 반드시 "퀴즈 내줘", "밸브 압력 알려줘", "오늘 뭐 했어" 등 주요 시나리오 직접 테스트

### 프롬프트 수정 효과 측정 방법
```
수정 전: python obsidian_agent.py "테스트 질문"
수정 후: python obsidian_agent.py "동일 테스트 질문"
→ 출력 비교
```

### 규칙 추가 시 형식
```
[규칙 이름]
트리거 조건: ~~~일 때
행동: ~~~하라
예외: ~~~는 적용 안 함
```
