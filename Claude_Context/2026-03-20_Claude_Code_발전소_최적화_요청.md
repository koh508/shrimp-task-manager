# Claude Code 요청: 발전소 시스템 과부하 및 토큰 사용량 최적화

## 요청 내용

1.  **(지도 확인)** 먼저 `SYSTEM/project_map.md`를 읽고 구조를 파악해.
2.  **(범위 한정)** 이번 작업은 '발전소' 시스템의 **토큰 사용량 최적화 및 과부하 방지** 기능 구현에 한정된다. 구체적으로는 **프롬프트 캐싱 시스템, 에이전트 워크플로우 최적화 도구, 토큰 사용량 모니터링 대시보드** 개발에 집중한다. 다른 파일이나 시스템 로직은 건드리지 마.
3.  **(작업 원칙)** 목표는 **API 호출 비용을 절감하고 시스템 안정성을 높이는 것**이야. 단, **기존의 핵심 자동화 흐름(Gemini 답변 수신, OneDrive 저장, 코드 정제, 옵시디언 답변)은 유지**해야 하는 게 원칙이야. Claude API의 `thinking.display: "omitted"` 설정 및 Gemini의 암시적 캐싱 기능을 적극 활용하는 방안을 포함해줘.
4.  **(결과 예측)** 로직이 꼬이지 않게 사전에 어떻게 수정할지 3줄로 계획을 먼저 말해줘. 계획이 맞으면 내가 진행하라고 할게. 코드를 출력할 때는 수정된 블록만 보여줘.

## 토큰 절약 팁 반영

*   **`.claudesignore` 파일:** 클로드 코드를 실행하는 루트 폴더에 `.claudesignore` 파일을 생성하여 불필요한 파일 스캔을 차단합니다.
*   **컨텍스트 압축:** 필요한 경우 `python SYSTEM\engine\onew_context_compressor.py` 스크립트를 사용하여 핵심 로직만 남긴 `.min.md` 파일을 참조하도록 지시합니다.

클로드코드의 답변:
온유가 존재하지 않는 "발전소" 시스템에 대한 작업을 Claude Code에게 시키는 프롬프트를 만든 겁니다. engine/ 폴더에
  strategy.py, task_planner.py가 있긴 하지만 "발전소"라는 시스템은 현재 운영 중인 게 아닙니다.

  더 큰 문제는 아까 onew_system_prompt.md에 추가한 중복 방지 로직인데:

  ▎ "프롬프트 작성 전 search_vault('작업일지')로 완료 항목 확인"

  온유가 이걸 안 했거나, 작업일지 검색을 했더라도 존재하지 않는 시스템에 대한 작업인지 판단을 못 한 겁니다.

  실질적으로 이 프롬프트에서 진짜 쓸모 있는 건:
  - .claudesignore 생성 — 이건 실제로 해볼 만함

  나머지 (프롬프트 캐싱 대시보드, 워크플로우 최적화 도구)는 온유 시스템에 당장 필요한 게 아닙니다. Gemini API 비용은
  이미 onew_budget.py로 관리 중이고요.

  .claudesignore만 만들까요?

> .claudesignore 할 경우 작업일지나 작업에 대한 사항은 일부 온유가 알지 못하는 거 아니야?

● .claudesignore는 Claude Code(저) 전용입니다. 온유랑은 무관합니다.

  - 온유 지식 경로: Vault 파일 → 임베딩 DB(onew_pure_db.json) → search_vault() RAG 검색
  - .claudesignore 영향 범위: Claude Code가 시작할 때 자동으로 스캔하는 파일 범위만 제한

  둘은 완전히 별개라 .claudesignore 만들어도 온유가 작업일지 못 읽는 일은 없습니다.

  단, 현재 Claude Code의 작업 디렉토리가 C:\Users\User(홈)이라 Vault 파일을 자동 스캔하지도 않습니다. 즉 .claudesignore
  만들어봤자 지금 환경에서 효과가 없습니다.

  결론: 온유 프롬프트에서 제안한 항목들 중 지금 당장 실질적으로 할 게 없습니다. 이 프롬프트는 무시하는 게 맞습니다.
